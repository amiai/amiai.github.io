Building a Design System

Project Overview

Team

Scrum team consisted of 3 software engineers, 1 QA engineer, 1 designer, and 1 product manager. Design team consisted of 3 designers, where I led the Design Systems project as the designer assigned to the scrum team.

Role

As the lead designer for this project, I was responsible for the creation of the Myagi Design System and also to collaborate closely with engineers in order to help build Storybook where code components were stored.

Tools

  • Figma
  • Miro
  • Confluence
  • Jira
  • Storybook
  • GitHub

Problem Statement

The accumulation of technical and design-debt, and the lack of documentation for the legacy product has resulted in inconsistencies across the product and the teams' shared language.

Objectives + Goals

Our primary objective for this project was to build an ongoing design system for a multitude of reasons. There are many benefits to having a design system. In particular, due to tech and design-debt, the team believed that it would advantageous as it would enable more efficient team collaboration, including cross-functional collaboration where having a shared vocabulary would allow two different departments to communicate expectations and objectives clearly.

Primary Goal

  • To create a 1:1 replication between code and design components

Secondary Goals

  • To streamline the team and company's shared vocabulary
  • To ensure alignment between Design and Engineering teams (i.e. Naming Conventions, Components)

Deliverables

  1. Figma file that contains all components (Designers and engineers can access)
  2. Storybook repository

Design Approach

Our team took on a more lean design approach due to limited time and resources. This involved utilising Tailwind's colour tokens, and reducing time spent on determining specific pixel and sizing guidelines.

As this project was occurring during the same time as the Frontend Refactor project, we would audit components for each page being refactored. After auditing, we would redesign components based on technical constraints and user research (See Frontend Refactor project for more information). Components would be designed and added into our Figma library, and once that was complete, engineers would add that into Storybook. Our audits took on a holistic approach as these components would be used in multiple pages, and not just a singular one. For example, we wanted to ensure that our 'Calendar' component could be used and easily accessible on more than just one page.

Product Users

Our primary target users for this design system were designers and engineers as our main objective was to create more efficient team collaboration and workflow between these two teams. However, the design system would also benefit other users which we have deemed our secondary target users.

Primary Users

  • Design Team
  • Engineering Team

Secondary Users

  • Sales and Marketing Team
  • Product Team
  • Support Team
  • Customers
  • The Business (As a whole)

Brand Guidelines

As a part of our objective to create an ongoing design system, it was critical that we documented information regarding not only our components and naming conventions, but also our brand guidelines. As our brand guidelines had already been determined prior to this project, it required collaboration with Sales and Marketing. In particular, the Marketing team was very knowledgeable about the Myagi's brand guidelines due to their frequent usage of it. As a result of collaboration with these teams, we were able to gauge guidelines ranging from tone of voice, colours, logos, and also infographics. Cross-functional collaboration was critical to this project as the design system's intended users include the entire company and the different departments within it.

Storybook + Figma Library

Storybook

Storybook is a development environment tool that allows engineers to document, create, and test UI components.

After components have been handed over to engineers and built, the components are then added into Storybook. Storybook is mainly used for the Engineering team's reference on the components, however, it was also used within the Design team to see how certain components would look and feel when a user interacts with it.

As seen in the examples below, Storybook allows the team to interact with the component by adjusting details such as:

  • Labels
  • Variants
  • Placeholders
  • Size
  • Toggling Full Width
  • Toggling Disabled

It was critical that information on components stored in Storybook were a 1:1 replication of the components in Figma. This would ensure that both the Design and Engineering team's understanding and knowledge of components were aligned.



Advantages of Storybook

  • Displays components available
  • Able to view components on different screen sizes
  • Quick access during development
  • Can store usage references and documentation
  • Interactive
    • Engineers and designers are able to see how a component will behave when there is user interaction
  • Can test components
    • Accessibility + colour filter features
  • Code references allow for more efficient development when implementing components
  • Designers can easily access Storybook to view how components works
    • Allows for review and comparison to mockups
    • Designers can stay updated with UI components on development side
    • Less follow-up questions and waiting for message updates from engineers

Outcome

Success Metrics

  • The design system is being used by designers
  • The design system is being used by engineers
  • Designers cut down time to make changes and updates to designs and handover
  • Consistent interface implemented on the new frontend
  • A 1:1 replication between code and design components

While the team did not have quantitative success metrics, we discussed what success would look like for this project prior to its initiation. The success metrics listed above were achieved. However, as a design system is always scaling, it would eventually require the team to implement a more tangible and quantitative form of measuring success. The team took on a more lean approach due to limited time and resources.


Final

The main objective for this project was achieved, which was to create a 1:1 replication between code and design components. This manifested in the form of a Figma file containing all our components, and a Storybook repository.

Reflection

What do you think was done well?

After building a 1:1 replication between Storybook and Figma components, we were able to handover designs more smoothly due to both the Design and Engineering team's shared vocabulary. Utilising Storybook in this way enabled us to work and collaborate closer and more efficiently with engineers. Having a shared vocabulary also allowed for more seamless discussions during meetings. For example, when deciding the most ideal component for a particular page, it was much easier to discuss as we all had the same understanding of what a radio button or a toggle would look like. Not only was this beneficial for Design and Engineering teams, but the design system was used by the Product team as well. This created unified collaboration between Product, Design, and Engineering teams as a whole.


What could have been improved?

Due to limited time and resources, it was challenging to build a more expansive design system. There are many aspects that tie into a design system, however, the team was unable to develop it further. Only a small portion of the design system was developed, including naming conventions, brand guidelines, component library, colour guidelines, and typography guidelines.


What would I do differently if I were to approach the same problem again?

If I were to approach this project again, I would ensure that the project to build and develop an ongoing design system has a higher priority. As the design system's primary target users were designers and engineers, and not customers, it was less of a priority compared to customer-facing projects. A design system has multiple advantages for the business, so I believe it would beneficial to gain more support for this project from the company. This could be done through a presentation which details reasons as to why this project would be effective and favourable for the business, and not just for the Design and Engineering teams.
